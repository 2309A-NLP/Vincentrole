# 角色扮演系统 — 优化总结

> 基于完整代码审查（2026-05-29），涵盖 main.py、rag_core.py、storage.py、embeddings.py、hybrid_retrieval.py、config.py、persistence.py、app.py 等 15+ 核心文件。

---

## 一、安全风险（高优先级）

### 1. config.py 硬编码 DeepSeek API 密钥
- 第 36 行：`LLM_API_KEY = "sk-9bf...0ef5"` — 密钥明文写在代码里
- 建议：改用环境变量 `os.getenv("DEEPSEEK_API_KEY")`，并在启动时校验非空

### 2. SECRET_KEY 使用占位符
- 第 40 行：`"replace-this-secret-before-production"` — 生产环境必须更换为随机字符串

### 3. ADMIN_TOKEN 可空
- 第 42 行：默认为空字符串，此时 `/api/admin/*` 接口返回 503
- 建议：在启动日志中明确警告未配置管理员令牌

### 4. 无速率限制
- 所有 API 接口（登录、注册、聊天）未设置限流
- 攻击者可暴力枚举密码或耗尽 LLM API 费用

### 5. 无请求参数长度校验
- 聊天接口的 `query` 字段无最大长度限制
- 建议设置 `max_length=4096` 防止极端输入

---

## 二、架构与稳定性（高优先级）

### 1. MilvusManager.__init__() 每次启动删除数据库
- `storage.py:59-64`：启动时 `utility.drop_collection()` 再重建
- 严重问题：服务重启会丢失所有已入库的知识库数据
- 建议：改为只创建不存在的集合，或提供配置项控制是否重建

### 2. 全局单例模块级初始化顺序耦合
- `main.py:63-65`：`from rag_core import chat_service` → `from storage import milvus_db` → `from embeddings import embed_engine`
- 这些模块在 import 时就会初始化（连接 Milvus、加载 BGE-m3 模型）
- 建议：改为懒加载或显式调用 `init()` 方法，便于测试和错误处理

### 3. BGE-m3 模型路径硬编码
- `config.py:11`：`LOCAL_BGE_M3_PATH` 硬编码为绝对路径
- 换机器或换用户就不可用，建议改为相对路径 + 环境变量覆盖

### 4. 前端 app.py 3223 行 — 单文件过于庞大
- 前端逻辑、CSS 样式（~450 行）、HTML 组件全部塞在一个文件里
- 建议拆分为：`components/`（可复用 UI 组件）、`styles.py`（CSS）、`api.py`（后端请求封装）

### 5. Redis 短期记忆和 SQLite 长期记忆存储重复
- `main.py` 通过 `app_store.save_message()` 存到 SQLite
- `rag_core.py` 通过 `redis_mem.add_message()` 存到 Redis
- 两个存储的数据重叠，增加了维护成本

## 三、性能优化（中优先级）

### 1. BGE-m3 1024 维向量较重
- 1024 维稠密向量在检索时计算量大，M4 上检索延迟会偏高
- 如果知识库规模增大（>10 万条），建议评估改用 768 维模型（如 bge-base-zh-v1.5）

### 2. 多查询变体并行检索的冗余
- `rag_core.py:143`：对前 3 个查询变体分别做 `embed_engine.encode()` + Milvus search
- 3 路稠密 + 1 路 BM25 = 4 路召回，再用 RRF 融合
- 实际效果上，对中文医疗问答来说查询变体带来的收益有限，可考虑减少为 1-2 路

### 3. 聊天接口无流式响应
- `rag_core.py:318-323`：使用 `stream=False` 等待完整回复
- 用户需要等待 LLM 完整生成后才看到回答，体验差
- 建议：后端支持 SSE streaming，前端逐步展示

### 4. 重复加载 CrossEncoder 模型
- `embeddings.py:32-37`：全局单例加载 BGE-Reranker-v2-m3
- 每次请求都使用同一个实例，但 `.predict()` 内部可能触发设备同步
- 如果 QPS 较高，可考虑用 asyncio 线程池隔离

### 5. 无缓存层
- 重复的用户问题每次都走完整 RAG 流程（编码 → 检索 → 重排 → LLM）
- 建议对高频问题添加简单缓存（如 Redis 缓存 5 分钟内相同问题的回答）

## 四、代码质量（中优先级）

### 1. 过时注释大量存在
- `config.py:26-28`：注释掉的 Ollama 配置
- `embeddings.py:109`：悬挂注释 `#复杂pdf的解析`
- `rag_core.py:61-62`：注释掉的 `api_key="EMPTY"`
- `test_rag_flow.py:34-35`：同样的注释
- 建议：统一清理，只保留当前生效的配置说明

### 2. eval.py 近乎空壳
- 只有 16 行骨架代码，不可用
- 要么补全为可运行的 RAGAS 评测脚本，要么删除

### 3. test_rag_flow.py 缺少依赖
- 第 8 行：`from langchain_huggingface import HuggingFaceEmbeddings` — 未列在项目依赖中

### 4. 缺少 requirements.txt
- 未找到依赖清单文件，新人接手需要逐个排查 import

### 5. 类型提示不完整
- `rag_core.py:340` 的 `chat()` 方法参数 `history_messages` 缺少类型标注
- `text_chunking.py:17` 的 `split_text()` 参数无类型提示

### 6. `_password_to_bytes()` 截断逻辑可能引起兼容问题
- `main.py:206`：只取密码前 72 字节，bcrypt 本身有这个限制
- 但用户设置的 8-20 位密码不会有此问题，这段代码可以简化

## 五、可观测性（低优先级）

### 1. 缺少请求时延监控
- 没有 Prometheus metrics 端点
- 建议：在 FastAPI 中添加中间件记录每个请求的 method/path/status/duration

### 2. 健康检查粒度不够
- `/health` 只返回 "models": "loaded" 字符串，不验证 Milvus 和 Redis 的实际可用性
- 建议：分别 ping Milvus 和 Redis，返回细粒度状态

### 3. 无日志采样
- DEBUG 级别的日志在 QPS 较高时会很多
- 建议：关键路径用 INFO，检索细节用 DEBUG 且考虑采样

---

## 六、部署与运维（低优先级）

### 1. 缺少 Dockerfile
- 后端直接 `python main.py` 启动，缺少容器化方案
- 建议提供 Dockerfile + docker-compose.yml（编排后端 + Milvus + Redis）

### 2. 无启动校验
- 启动时不检查 Milvus/Redis 是否可达，不检查 API key 是否配置
- 建议在 startup event 中做完整的健康检查

### 3. 无优雅关闭
- `uvicorn.run(app)` 不带 graceful shutdown 配置
- 建议添加 lifespan handler 做资源清理（关闭 Milvus 连接、Redis 连接）
