# 工单1 总结与优化点

## 一、项目概况

**项目名称：** 基于PDF文档的智能问答系统（RAG架构）
**技术栈：** Python + Streamlit + FAISS + BGE + DeepSeek/OpenAI
**核心流程：** PDF解析 → 文本分块 → 向量化 → FAISS索引 → 语义检索 → LLM生成

---

## 二、模块总结

| 模块 | 文件 | 职责 |
|------|------|------|
| PDF解析 | `pdf_parser/parser.py` | 支持 pymupdf / pypdf 双引擎，文本提取与清洗 |
| 文本分块 | `pdf_parser/chunker.py` | 智能分块（500字/100重叠），保持段落/句子完整性 |
| 向量嵌入 | `knowledge_base/embeddings.py` | BGE-base-zh-v1.5，支持 TF-IDF 离线回退 |
| 向量存储 | `knowledge_base/vector_store.py` | FAISS IndexFlatIP，支持持久化/加载 |
| Query理解 | `qa_engine/query_understanding.py` | 意图识别、实体提取、查询重写、多语言支持 |
| 检索器 | `qa_engine/retriever.py` | 向量检索 + 结果排序过滤 + 上下文构建 |
| LLM生成 | `qa_engine/generator.py` | 多后端（DeepSeek/OpenAI/Ollama），模拟模式兜底 |
| 主控制器 | `qa_engine/orchestrator.py` | RAGSystem 串联全流程，支持 RAG vs 纯LLM 对比 |
| 前端界面 | `app/ui.py` | Streamlit 聊天界面 + 侧边栏配置 + 语音输入 |

---

## 三、现有优势

1. **双引擎PDF解析** — pymupdf 优先，pypdf 兜底，兼容性好
2. **智能分块策略** — 按段落→句子→字符三级切分，保留语义完整性
3. **TF-IDF离线回退** — 无法下载 BGE 模型时自动降级，保证可运行
4. **Query理解** — 意图识别 + 实体提取 + 查询重写，提升检索精度
5. **多LLM后端** — DeepSeek / OpenAI / Ollama 灵活切换，演示模式兜底
6. **对比模式** — RAG vs 纯LLM 并排对比，直观展示 RAG 价值
7. **语音输入** — Web Speech API，无需额外依赖
8. **缓存机制** — FAISS索引持久化，避免重复解析

## 四、优化点（按优先级排序）

### P0 — 核心质量

#### 1. Embedding 模型路径硬编码问题
- **现状：** `config.py` 中 `EMBEDDING_CONFIG["模型名称"]` 直接写 `"BAAI/bge-base-zh-v1.5"`，依赖 HuggingFace 镜像下载
- **问题：** 国内网络不稳定，首次加载可能极慢或失败
- **建议：** 改为本地快照路径（如 `BAAI/bge-base-zh-v1.5` 的本地缓存路径），或在 `requirements.txt` 旁加 `setup.sh` 一键下载模型到本地

#### 2. LLM API Key 泄露风险
- **现状：** `config.py` 第72行明文写入 `API密钥: "sk-13c...6164"`
- **问题：** 代码一旦提交到 Git，密钥即泄露
- **建议：** 改用环境变量 `os.environ.get("DEEPSEEK_API_KEY", "")`，或 `.env` 文件 + `.gitignore`

#### 3. FAISS 索引维度不校验
- **现状：** `VectorStore.load()` 加载索引后直接赋值 `self.dimension = self._index.d`，但如果换了 embedding 模型（维度变化），旧索引不会报错，检索结果全部错误
- **建议：** 加载时校验维度一致性，不匹配则提示重建索引

### P1 — 检索质量

#### 4. 分块策略可优化
- **现状：** 固定 500 字/100 重叠，对表格、列表等结构化内容不友好
- **建议：**
  - 表格内容整块保留，不硬切
  - 支持按语义分块（如 SemanticChunker）
  - 分块大小可根据文档类型自适应

#### 5. Query 理解能力有限
- **现状：** 基于关键词匹配的意图识别，无 LLM 参与
- **问题：** 复杂问句（如"公司近三年营收增长率和净利润率分别是多少？"）无法正确分解
- **建议：** 引入 LLM 做 query decomposition，或接入 RAG Fusion 多路检索

#### 6. 缺少重排序（Reranker）
- **现状：** 直接用 FAISS 向量相似度排序，无二次精排
- **问题：** 向量召回的 Top-K 中可能有噪音片段
- **建议：** 加入 cross-encoder reranker（如 `bge-reranker-base`），对 Top-K 结果精排后再送入 LLM

#### 7. 检索策略单一
- **现状：** 纯向量检索，`config.py` 中 `启用混合检索: False` 已预留但未实现
- **建议：** 实现 BM25 + 向量的混合检索（Hybrid Search），互补关键词精确匹配与语义理解

### P2 — 系统健壮性

#### 8. 错误处理不够精细
- **现状：** 多处 `except Exception as e` 笼统捕获，用户只看到"失败"
- **建议：** 区分网络超时、API 限流（429）、模型不可用等场景，给出针对性提示

#### 9. 缺少并发与超时控制
- **现状：** LLM 请求 `timeout=30`，但无重试机制（`max_retries=3` 已定义但未使用）
- **建议：** 实现指数退避重试，处理 DeepSeek 429 限流

#### 10. 无日志系统
- **现状：** 全用 `print()` 输出日志
- **建议：** 改用 `logging` 模块，支持日志级别、文件输出、结构化日志

### P3 — 用户体验

#### 11. 首次加载体验差
- **现状：** 用户点"加载文档"后要等很久（模型下载 + PDF解析 + 向量化），期间只有一个 spinner
- **建议：** 分步进度条（解析中→分块中→向量化中→构建索引），或后台预加载模型

#### 12. 对比模式结果展示粗糙
- **现状：** RAG 和纯 LLM 回答在两个 Tab 中展示，无法直观对比差异
- **建议：** 改为左右分栏并排展示，高亮差异部分

#### 13. 缺少引用溯源
- **现状：** 回答中不标注引用了哪些原文片段
- **建议：** 实现 citation 机制，答案中标注 `[来源: 第X页]`，点击可展开原文

#### 14. 不支持多轮对话
- **现状：** 每次提问都是独立的，无上下文记忆
- **建议：** 引入对话历史窗口，支持指代消解（如"它的股东是谁？"中的"它"）

### P4 — 工程规范

#### 15. 无单元测试
- **现状：** 各模块有 `if __name__ == "__main__"` 快速测试，但无 pytest 测试用例
- **建议：** 为每个模块编写单元测试，重点覆盖分块逻辑、检索准确性、LLM 回退

#### 16. 无 requirements.txt 版本锁定
- **现状：** `requirements.txt` 存在但未检查是否锁定了版本
- **建议：** 使用 `pip freeze` 锁定版本，或改用 `pyproject.toml` + `poetry`

#### 17. 项目结构可优化
- **现状：** 实际代码在 `研发/` 子目录下，外层是空壳
- **建议：** 将 `研发/` 内容提升到项目根目录，消除不必要的嵌套

## 六、一句话总结

> 工单1 搭建了一个功能完整的 RAG 问答骨架，核心流程跑通，演示模式可兜底。主要优化方向是**检索质量**（Reranker + 混合检索）和**工程健壮性**（安全、测试、日志），其次是**用户体验**（多轮对话、引用溯源、进度反馈）。
